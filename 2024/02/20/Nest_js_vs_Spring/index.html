    Nest.js와 Spring 비교

    2024, Feb 20

    배경

    NestJS가 어떤 프레임워크를 공부하는 과정에서 우아콘 세션을 듣게 되었는데, Spring과 함께 비교하는 내용이 잘 정리되어 있다고 생각하여 정리하게 되었습니다.

    들어가기 앞서

    기존 사용하던 Spring은

    • 많은 레퍼런스, 모범 사례가 있어 안정적인 운영이 가능
    • 새로운 엔지니어가 합류해도 균일한 코드 퀄리티 유지 가능
    • 멀티스레드 기반 (CPU 집약적인 작업)
      • 장점
        • 하나의 세션에 신뢰성이 높음
      • 단점
        • 온디맨드 트래픽이 많은 경우, 준비해야 하는 시스템 자원이 많음
          • Serverless나 빠르게 Scale-out 해야 하는 상황에서는 Cold-start 시간이 오래 걸림

    어떤 요구사항이 있었나

    • 어플리케이션, 비즈니스 로직의 고도화를 위한 도구가 필요했음
    • 많은 엔지니어가 함께 개발할 수 있어야 했음
      • 대중적, 일정 수준 이상의 생태계가 구성되어야 했음
    • 생산성과 안정성
      • 현대의 복잡한 요구 사항을 빠르게 만족시키고, 안정적으로 구현해야 했음
      • 개발 사이클에서 컨벤션 또는 가이드가 되어 있어 예측 가능해야 함

    과정

    요구사항을 해결하기 위해 선택한 JS

    간결함과 실용주의를 지향

    • 싱글스레드로 Blocking I/O를 효과적으로 처리할 수 있음
      • Ruby-이벤트 머신, Python-트위스트 같은 시스템 설계에 영향을 받아 설계가 유사
      • 이벤트 루프를 통해 시스템 커널(OS)로 작업을 넘기는 방식으로 Non-blocking I/O를 구현
        • 현대 커널 환경은 멀티스레드이기 때문에 백그라운드에서 다수의 작업이 실행
        • 완료되면 Node.js의 Callback Queue에 추가하는 방식으로 설계
    • 멀티스레드에 비해 많은 I/O 작업을 처리하는데 필요한 자원이 적음 (I/O 집약적인 작업)
      • 요청이 많고 가벼운 REST API 환경에 적합
      • 작은 여러개의 서비스 아키텍처에 적합

    Typescript 특징

    • 최근 5년간 3배 이상 성장하고, 약타입인 JS 대비 안정적이고 효율적인 개발이 가능
    • 런타임 시에는 자바스크립트로 동작하기 때문에 타입을 엄격하게 관리할 수 없음
      • 약타입과 강타입 중간에 위치
      • 강타입 장점
        • 결과의 일관성을 보장
        • 컴파일러 단계에서 에러를 감지
          • 코드 레벨에서의 문제를 조기에 발견
      • 약타입 장점
        • 생산성이 높아서 개발 속도가 빠름
    • 타입을 엄격하게 관리한다면 강타입만큼 높은 코드 퀄리티를 유지할 수 있음
      • 프레임워크에서 타입 스크립트를 완벽하게 제공하거나 API, DB 등 3rd party service 로부터 전달받는 데이터의 타입이 명확하게 관리되어야 함

    CPU 집약적인 작업, I/O 집약적인 작업이란

    CPU 집약적인 작업 (CPU-bound)

    • 중앙 처리 장치(CPU)의 계산 능력이 필요한 작업(계산, 논리 연산, 알고리즘 실행)
    • 계산 복잡도가 높은 수학적 계산
      • 대규모 행렬 곱셈, 복잡한 수학적 모델링, 알고리즘 최적화 작업 등
    • 3D 그래픽 렌더링
      • 3D 게임 또는 애니메이션의 그래픽 렌더링 작업은 복잡한 계산으로 인해 CPU의 계산 능력이 필요
    • 암호화 및 복호화 작업
      • 데이터베이스에서 암호화된 데이터를 추출하거나, 전송된 데이터를 복호화하는 작업
    • 심층 학습 모델의 학습 단계
      • 딥러닝 모델을 학습시키는 작업은 대용량의 수치 연산으로 인해 CPU의 계산 능력이 필요

    I/O 집약적인 작업 (I/O-bound)

    • 주로 입출력 작업에 의해 제한되는 작업
    • 파일 읽기/쓰기 작업
      • 대용량 파일의 읽기 또는 쓰기, 데이터베이스에 대량의 데이터를 삽입하는 작업
    • 네트워크 통신
      • 원격 서버와의 데이터 교환, 웹 서비스로의 API 호출
    • 데이터베이스 쿼리
      • 복잡한 데이터베이스 쿼리나 대량의 데이터를 가져오는 작업
    • 사용자 입력 처리
      • 대화형 응용 프로그램에서 사용자 입력, GUI 요소의 표시와 상호 작용

    Node.js 진영의 프레임워크

    Express

    • 확장성을 지향
    • 장점
      • 불필요한 간섭을 하지 않음
      • 자유도가 높음
      • DB, Logging, Cache, Messaging, DI, Validation 등 자신에게 맞는 라이브러리를 선택
    • 단점
      • 시스템이 커지고 협업이 늘어날수록, 컨벤션 통합에 대한 비용과 시스템 복잡도가 커짐
      • 새로운 기능을 추가할 때마다 수많은 시작 코드와 테스트, 최적화, 검증이 필요
      • 타입스크립트를 지원하지 않음
      • 선택한 라이브러리 간 버전 관리 이슈가 발생하거나 deprecate 될 수 있는 문제가 있음

    nest.js

    • angular에 영향을 받아, Spring에 더 가까운 Opinionated 프레임워크 형태로 구현
      • 패턴을 강제하는 대신 다양한 기능을 제공하는 완전 관리형 프레임워크
      • 지향하는 철학과 개념이 Spring과 유사하여, 코드 레벨에서 유사한 모습을 보임
    • 장점
      • 이해하기 쉬운 코드 베이스로 작성
      • 일관된 개발로 예측 가능
      • 다양한 확장 패턴이 고려되어 있어, 요구 사항에 맞게 활용이 가능
    • 단점
      • 프레임워크가 지향하는 목적과 요구 사항이 일치하지 않을 경우, 프레임워크 레벨의 수정이 필요
      • 강제된 규칙을 벗어나기 어려움


    비교

    Nest.js와 Spring

    공통점

    지향하는 철학

    • OOP (Object Oriented Programming)
    • FP (Functional Programming)
    • FRP (Functional Reactive Programming)

    사용하는 개념

    • 어노테이션, 데코레이더
    • IoC Container, DI
    • Layered Architecture
    • ORM을 활용한 Repository Pattern

    차이점

    Spring

    • 실행 기반으로 공통 함수를 유지 및 보수하기 용이 (Maintain)
    • 다중 스레드 처리 지원 및 멀티스레딩에 적합한 구조 (Multi-threading)
    • 컴포넌트들이 성숙하게 발전하고 있어 안정적인 사용이 가능 (Mature components)
    • 공통 함수 실행에 중점을 둔 실행 기반 (Running base of common functions)
    • 대규모 및 복잡한 프로젝트에 대한 지원 및 대응이 가능 (Support Large More Complex Projects)

    Nest.js

    • 빠른 실행 및 응답 속도 (Faster)
    • 입출력 처리에 있어 효율적인 핸들링 (Handling I/O)
    • 공통 기능을 모듈화하여 효율적으로 사용 (Modularize some common functions)
    • 주로 웹 애플리케이션 서버로 사용 (More inclined to Web Application Server)


    Nest.js의 구조

    Nest.js에서 하나의 요청에 대한 라이프 사이클

    1. Request
    2. Middleware
    3. Guard
    4. Interceptor
    5. Pipe
    6. Controller
    7. Service
    8. Controller
    9. Interceptor
    10. Filter
    11. Response

    Modular Layered Architecture를 기반으로 사용

    • Module 단위로 도메인과 서비스, 라이브러리를 분리하고 관리, Module 내에서 Layer를 분리
      • 확장에 유리
      • 복잡한 비즈니스를 쉽게 관리
    • 라이브러리는 준비만 되어 있다면 별도 개발 없이 쉽게 추가 가능
      • 안정적이고 빠르게 요구 사항을 구현할 수 있음

    하나의 요청에 대한 라이프 사이클을 Authenticate, Validation, Error 처리 등 각 단계별로 관리

    • 코드의 가독성, 유지보수성이 향상
    • 강타입만큼 안전한 서비스를 유지할 수 있음
    • 각 단계의 처리는 특정 부분뿐 아닌 코드 전반에 적용하여 사용 가능
      • e.g. Validation: class-validator, class-transformer를 이용해서 전달받는 데이터의 타입을 관리

    설정 관리를 다양하게 하는 자유가 있음

    • 설정값들이 추상화되어 모듈 단위를 벗어나서 관리됨
    • 값의 타입이 추상화되어 관리됨

    트랜스포트 레이어를 별도로 관리되며 변경 가능

    • Http 뿐 아니라, gRPC, Web-socket, I/O 등 다양한 통신 프로토콜을 쉽게 적용할 수 있도록 지원
    • 트랜스포트 레이어의 변경을 통해 Request, Response의 관리를 통신 레이어에서 명확하게 할 수 있도록 지원
      • 비즈니스 로직이 프로토콜의 영향을 받지 않음
      • 하나의 비즈니스 로직으로 다양한 프로토콜을 처리 가능

    독립적으로(Stand-alone) 동작하는 Lambda 형태도 지원


    Nest.js 환경의 개발을 위한 고민

    최소 품질을 보장하기 위한 방법

    하나의 코드 베이스로 다양한 환경을 제공

    • 트랜스포트 레이어 별로 필요한 에러, 핸들링 등 요구사항은 컨트롤러에서 처리
    • 이벤트 프로토콜과 같은 전사 표준 형태를 템플릿에 규정

    공통 템플릿(보일러 플레이트)를 제공

    • 전사 표준 환경을 지키면서 개발 가능
    • 비즈니스 영역을 분리

    CI/CD 환경도 템플릿에 포함

    • 인프라 설정 ~ 배포에 대한 관리 부담을 해소

    동일한 코드 형태의 개발

    • lint rule을 합의하고 개선하며 개발 컨벤션을 강제

    높은 커버리지의 테스트 환경

    • 테스트를 통해, 코드의 변경이 기존 시스템의 의도를 해치지 않았다는 것을 보장받을 수 있음
    • 높은 커버리지가 ‘버그가 없다’의 의미는 아니지만, 의도에 맞게 동작하고 있다는 정도의 보장을 의미

    @nestjs-library/crud

    • CRUD에 대한 반복적인 작업을 어노테이션 정의를 통해 단순화
    • 어노테이션 옵션을 통해 필요한 것만 활성화하거나 Interceptor, decorator 등을 커스텀
    • 참고 Repo

    @nestjs-library/config

    • nest.js의 자유로운 config 관리로 인해 하나의 설정 값이 다른 의미로 사용되거나 전역으로 사용되지 않는 설정을 관리하기 위한 비용이 발생
    • config를 서비스(class)로 구성하여 모듈 단위로 임포트하여 싱글톤 프로바이더로 설정을 관리하는 방식
      • 서비스의 포맷에 맞게 관리하고 매핑
      • 각 설정에 대해 클래스 별로 명확하게 오너십을 가지도록 함
    • 읽기 전용으로 수정이 불가하여 값에 대해 보호
    • 런타임 때, 의도적으로 값을 변경하는 경우는 validation 로직을 포함한 함수의 형태로 지원

    • 사용되고 있는 설정을 확인하는데 어려움이 있어, 별도로 문서화하는 기능을 구현

    • 참고 Repo
