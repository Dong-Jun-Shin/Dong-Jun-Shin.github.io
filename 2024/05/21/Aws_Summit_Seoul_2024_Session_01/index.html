    DB 마이그레이션을 하기 위한 네 가지 문제점

    [문제1 - 스파이크 트래픽과 효율적이지 않은 비용]

    [문제2 - 현재 구조는 부하가 전파될 수 있음]

    [문제3 - 운영에 대한 호환성]

    [문제4 - 마이그레이션 과정 중 다운타임의 최소화]

    [그 외, 이벤트 소싱과 AWS 연동 고려]

    문제점을 해결한 네 가지 방법

    [해결 방법 1, 2 - DynamoDB 기능 활용]

    • 온디맨드 모드와 프리워밍으로 스파이크 트래픽 대응
    • 프로비저닝 모드와 오토 스케일링으로 비용 문제 해결
    • DynamoDB로 옮기면서 독립적인 테이블로 설계하여 부하가 전파되는 것을 방지

    [해결 방법 3 - DynamoDB ACID 트랜잭션 기능을 활용한 형태의 테이블 전략]

    [원자적 연산과 집계 로직]

    • ManagerBadge를 얻기 위한 로직에서 트랜잭션을 활용해서 업데이트 하는 방법으로 적용하여 성능을 향상

    [데이터를 연산하는 로직을 다른 방법으로 접근]

    • 트랜잭션 활용 시, 생기는 잠금과 충돌 문제를 해결하기 위해 재시도와 멱등성을 활용
    • 문제가 발생한 작업은 재시도하도록 처리
    • 재시도를 하면서 생길 수 있는 문제는 Client Request ID로 체크해서 멱등성(동일한 결과)을 보장할 수 있도록 함

    [낙관적 잠금과 충돌에 대해]

    • Worker Thread를 병렬로 구성하고 Batch로 진행
    • 마이그레이션 버전업으로 이벤트를 발생시킴

    [해결 방법 4 - 병렬 스캔하는 방식과 Batch를 활용]

    DynamoDB Streams와 Local 활용 방법

    • 기존 방식

    [DynamoDB Streams를 활용한 이벤트 소싱]

    • 기존 방식과 다르게, API Server에서 직접 새로운 테이블에 쌓는 방식을 선택

    [DynamoDB로 실시간 마이그레이션하는 흐름]

    [테스트를 위한 로컬 환경 구축]

    • DynamoDBEmbedded는 In-memory에서 작동하기 때문에 속도 향상
    • 25배의 속도 개선을 경험

    [DynamoDBEmbedded API 활용한 방식]

    스파이크 트래픽을 처리하기 위한 Amazon SQS

    • 오토스케일링은 확장되기 전 순간적인 트래픽 처리를 하기에 부족함

    [AWS SQS가 필요한 이유]

    [AWS SQS를 활용하기 위해 고려한 사항]

    [트래픽 버퍼 처리 방법]

    • SQS는 완전 관리형 서비스이기 때문에 처리하는 속도가 충분한지 모니터링하면 됨

    [완전 관리형 서비스인 AWS SQS]

    [FIFO 큐 처리 제한으로 인한 적절한 키 디자인의 중요성]

    [FIFO 큐 처리 제한에 대한 공식 페이지 참고]

    • 처리 속도를 조절하기 위해 애플리케이션에서 직접 polling하는 방식을 선택

    [AWS SQS Consumer 처리 방식]

    [TPS 증가에 따라 Worker당 처리량 변화]

    • SQS 대신 DynamoDB Streams + Kinesis Data Streams를 활용해서 여러개의 Shard로 구성하는 방법도 있음

    [Scale-up하는 방법]

    • DynamoDB Streams + Kinesis Data Streams로 구성하게 된다면, 동일한 사용자에 대한 웹훅이 동시에 여러개 생겼을 때, 실패하게 됨

    [DynamoDB Streams + Kinesis Data Streams로 웹훅 처리한 경우]

    • SQS는 동일한 메시지 그룹에서 하나의 메시지만 처리하게 됨

    [SQS로 웹훅 처리한 경우]
