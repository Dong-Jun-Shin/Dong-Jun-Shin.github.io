    • AWS Graviton 마이그레이션 해야 할 이유

      • 난이도 분류를 통한 소요 시간 예측

      • 파이썬 ML 서버를 마이그레이션 한 후 최적화 과정
        • SIMD(json 개선 라이브러리)
        • Firefox의 profiler + Perf를 이용한 프로파일링
        • base64 -> Pybase64
        • Numpy 최적화
          • 아키텍처 방식에 맞는 타입 사용
          • 불필요한 연산 제거
        • 적절한 Memory allocator 사용
        • 최적화 결과 최대 27% 정도의 개선을 이룸
      • 데이터 파이프라인을 에뮬레이터로 마이그레이션 하는 방법
        • 마이그레이션 하기 위해 고려한 사항
          • 가능한 큰 변경없이 이전하기를 원함
        • 마이그레이션 방법 분석 및 비교
          • Native Graviton
          • QUS(QEMU)
          • Box64
        • QUS를 이용한 마이그레이션 과정 및 결과
          • 빠른 마이그레이션이 가능했지만, 실행 시간이 약 400% 늘어남
        • Box64를 이용한 마이그레이션 과정 및 결과
          • 소스 코드의 수정이 필요했지만, 실행 시간이 약 25% 늘어남
    • 후기
      • 마이그레이션은 오래 걸릴 수 있으니, 꼼꼼한 계획과 꾸준한 진행이 필요

    AWS Graviton 소개

    • 전문화를 통한 효율성 향상, 성능 최적화를 통한 비용 감소, 보안 문제 해결
    • Inferetina, tranium 등 AI 서비스를 위한 인스턴스 지원

    [AWS는 맞춤형 실리콘 서비스를 통해 다양한 문제를 해결 가능함]

    AWS Graviton 마이그레이션 해야 할 이유

    • Graviton은 에너지 소비를 줄이고 비용을 줄일 수 있도록 설계된 프로세서 제품군

    [AWS Graviton은 EC2 대비 더 좋은 가성비를 제공할 수 있음]

    • 2018년 처음 출시한 이후, 작년에 Graviton 4세대를 출시

    [Graviton은 4번에 걸쳐 업그레이드 되어 왔음]

    • 컴퓨팅 연산 향상, DDR5 메모리 지원, 전력 소비 감소, ML 연산 성능 향상

    [Graviton 3세대 특징]

    • 소켓당 코어 수 50% 증가
    • 시스템당 3배 정도 많은 코어, DRAM

    [이전 세대 대비 Graviton 4세대의 개선점]

    [마이그레이션을 통한 목표 설정]

    • 영향도와 난이도, 효과를 고려한 우선순위 선정
    • 먼저 중요도가 낮은 서비스들로 숙련도 향상을 도모
    • 그 다음으로 cpu 사용률이 높은 시스템 위주로 그래비톤 사용률을 높임

    [우선순위 설정]

    • 프로그램에 필요한 모든 라이브러리를 최종 실행 파일에 그대로 삽입하는 형태(static linked binary)는 쉬움
    • H/W나 OS와 같은 네이티브 애플리케이션와 호출하기 위한 인터페이스(JNI)를 사용하는 경우는 꽤 까다로움
    • 아키텍처 호환이 되지 않는(의존성이 큰) 라이브러리가 많은 경우 어려움

    • Java와 Python 서버를 마이그레이션하는 문제에 직면
