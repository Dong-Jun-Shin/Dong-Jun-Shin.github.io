    Summary

    • 파운데이션 모델 커스터마이징
      • 커스터마이징은 도메인 특화 용어 사용, 전문성 강화, 문맥 인식 능력 향상을 위해 필요함
      • 커스터마이징 방법에는 프롬프트 엔지니어링, 검색 증강 생성(RAG), 파인 튜닝, 모델 훈련이 있음
      • 각 방법은 품질을 향상시키지만, 복잡도, 비용, 시간도 증가함
    • 검색 증강 생성 (RAG) 개요
      • RAG는 외부 데이터를 검색해 모델 입력에 제공하여 증강된 응답을 얻는 기법임
      • 사용자의 과거 데이터를 활용해 개인화된 검색이 가능하고, DB나 API 호출로 데이터를 요약할 수 있음
      • 데이터 검색 방식으로 규칙 기반, 구조화된 데이터, 시맨틱 검색이 있으며, 임베딩은 단어 간의 의미와 관계를 벡터로 표현한 것임
    • Amazon Bedrock 지식저장소란?
      • Bedrock의 지식 저장소는 RAG 워크플로우를 End-to-End로 지원하는 완전 관리형 서비스임
      • S3와 같은 데이터 저장소 연동, 증분 업데이트, 다양한 파일 형식 지원 및 임베딩 모델 연동을 제공함
      • Retrieve API, RetrieveAndGenerate API를 이용해 임베딩 또는, 임베딩부터 LLM 응답 생성까지 처리할 수 있음
    • Advanced RAG (고급 기법)
      • Hybrid-Fusion, RAG-Fusion, HyDE, Parent Document, Re-ranker 등 여러 기법을 조합해 사용자의 쿼리에 대한 검색 정확도를 높일 수 있음
      • 다양한 검색 결과를 결합하고, 가상 문서를 생성하여 검색 범위를 확장함
      • 복잡한 문서를 임베딩하고 시맨틱 검색을 결합해 더 정확하고 유용한 정보를 제공함
    • Amazon Bedrock 지식저장소 및 RAG 고급 데모
      • Amazon Bedrock 지식저장소를 생성하고 데이터를 추가하여 RAG를 활용하는 데모임
      • LangChain과 SDK를 사용하여 지식저장소를 호출하는 코드를 제시함
      • 복잡한 문서를 임베딩하고 Advanced RAG 기법을 적용한 시스템을 시연함

    파운데이션 모델 커스터마이징

    • 커스터마이징은 도메인에 특화된 용어 사용, 전문성 강화, 문맥 인식 능력 향상을 위해 필요함

    [모델 커스터마이징의 필요성]

    • 커스터마이징하기 위한 방법으로 프롬프트 엔지니어링, 검색 증강 생성(RAG), 파인 튜닝, 모델 훈련 4가지가 있음
    • 각 방법에 따라 품질이 좋아지지만, 적용하기 위한 복잡한 정도, 비용, 시간도 함께 늘어남

    [모델의 커스터마이징 방법론]

    • 커스터마이징 방법은 특정 정보의 필요 여부, 실시간 추론의 필요 여부, 데이터의 복잡도에 따라 선택할 수 있음
    • RAG로 증강: 특정 정보이면서 문서와 같은 정적 데이터
    • 에이전트, 도구로 보강: 실시간 업데이트되는 데이터
    • 파인 튜닝: 복잡하거나 보편적이지 않은 도메인 데이터
    • 프롬프트 엔지니어링: 간단하고 보편적인 데이터

    검색 증강 생성 (RAG) 개요

    • RAG는 외부 데이터를 검색해서 모델의 입력에 제공함으로, 증강된 응답을 얻는 기법
    • 질문의 답이 있는 문서를 찾는 기술

    [검색 증강 생성이란]

    [검색 증강 생성의 한 줄 정의]

    • RAG는 외부 정보를 반영해서 답변의 정확도를 향상
    • 외부 정보로 실시간 데이터도 있고, 이를 활용해서 챗봇과 같은 기능을 강화하기에 적합
    • 사용자의 과거 데이터를 이용해서 개인화된 검색을 하는 것이 가능
    • DB나 API 호출을 통해 데이터를 요약할 수 있음

    [RAG 사용 사례]

    • 데이터를 검색하는 방법으로 규칙 기반, 구조화된 데이터, 시맨틱 검색이 있음
    • 규칙 기반은 정형화되지 않은 데이터에서 일치하는 키워드를 기준으로 검색하는 방법
    • 구조화된 데이터는 DB와 같이 정형화된 데이터에 쿼리문을 이용해서 검색하는 방법
    • 시맨틱 검색은 벡터 DB와 같이 의미론적으로 더 관련있는 데이터를 검색하는 방법

    [데이터를 검색하는 방식]

    • 임베딩은 단어 간의 의미와 관계를 숫자(벡터)로 표현한 것
    • 유사한 정도에 따라 데이터가 모이고, 군집의 형태로 구성됨

    [임베딩이란]

    • RAG는 연관성이 높은 데이터를 사용할수록 정확도를 높일 수 있기 때문에 검색하는 방식이 중요
    • 검색의 정확도를 높이기 위해 검색 방식이 중요하고, 의미론적 관계를 나타낼 수 있는 임베딩 방식이 적합함

    [RAG에서 임베딩이 중요한 이유]

    • 텍스트 임베딩을 위한 모델로 AWS에서 Amazon Titan 서비스를 사용할 수 있음

    [Amazon Titan 소개]

    • RAG는 사용자가 프롬프트를 입력하면 임베딩 모델을 통해 시맨틱 검색을 하고, 결과로 나온 컨택스트로 프롬프트를 증강시켜서 더 품질높은 응답을 어듬

    [RAG의 동작 흐름]

    • RAG는 아래와 같은 문제로 어려움이 있음
      • DB, 파일, 크롤링 등 여러 형태의 데이터 소스에 대해 관리가 필요함
      • 데이터 크기가 큰 경우 벡터로 임베딩할 때, 많은 컴퓨팅 자원과 시간이 필요
      • 새로운 데이터가 추가될 때마다 벡터 스토어에 갱신이 필요함 (추가에 대한 증분 업데이트)
      • RAG 시스템을 구축하고 유지하기 위해, 데이터 수집부터 결과 활용까지 여러 단계에 대한 구현이 필요함
      • RAG 시스템이 대규모 데이터 처리와 실시간 검색을 제공하고, 동시에 많은 처리를 요청할 수 있도록 확장성도 고려해야 함
      • 데이터 수집 ~ 결과 활용 단계 중 여러 작업에 대한 순차 처리, 병렬 처리에 대한 고려가 필요하고 시스템 구성 요소 간 의존성을 관리가 필요

    [RAG 구현을 위해 극복해야 할 기술적 문제들]

    Amazon Bedrock 지식저장소란?

    • Bedrock의 지식 저장소(Knowledge base)는 완전 관리형 서비스로 RAG 워크플로에 대해 End-to-End로 지원
    • 모델에 데이터 저장소를 안전하게 연결
    • 저작권 표시와 손쉬운 검색을 제공

    [Amazon Bedrock 지식저장소 소개]

    • S3와 같은 데이터 저장소 연동, 증분 업데이트 지원, 다양한 파일 형식 지원
    • 문서 청킹 전략 지원
    • 임베딩 모델 연동 제공
    • 다양한 벡터 스토어 지원

    [Bedrock 사용 시, 데이터 통합 워크플로 단계에 지원하는 기능]

    • Amazon Bedrock을 사용할 경우, 데이터 통합에 대한 자동화와 데이터 수집에 대한 완전한 관리가 가능

    [완전 관리형 서비스]

    • Bedrock 지식저장소는 RAG를 위한 외부 데이터 검색 및 생성 단계에서 벡터 스토어 관리와 임베딩 기능을 지원

    [Amazon Bedrock 지식저장소를 데이터 관리 부분에 적용]

    • 더 나아가, 지식저장소에서 제공하는 RetrieveAndGenerate API를 이용해서 임베딩 ~ LLM 응답 생성까지 처리할 수 있음

    [검색 및 생성 API 소개]

    • Retrieve API로 임베딩 및 저장소 검색까지만 처리할 수도 있음

    Advanced RAG (고급 기법)

    • Hybrid-Fusion: 외부 데이터를 검색할 때, 시맨틱 검색과 키워드 검색을 같이 사용하는 방법
      • 정확도는 올라가지만, 구현 및 유지보수의 복잡성 증가와 성능 문제가 발생할 수 있음
    • RAG-Fusion (Query Rewriting): 사용자의 쿼리에 대해 모델로 재작성하여 명확한 질문의 형태로 만들고 시맨틱 검색에 사용하는 방법
      • 구체적이지 않은 쿼리도 비교적 훌륭하게 답변을 생성하지만, 재작성 과정에서 결과가 왜곡되거나 처리 시간이 길어질 수 있음
    • HyDE (Hypothetical Document Embeddings): 쿼리를 분석하여 주제와 관련된 가상의 문서를 생성하고 임베딩한 후, 시맨틱 검색에 사용하는 방법
      • 검색 범위가 확장되어 다양성이 높아질 수 있으나, 가상의 문서를 생성하고 관리하는 것이 복잡할 수 있고 가상 문서에 실제 문서가 반영되지 못하면 신뢰성이 떨어질 수 있음
    • Rank-Fusion으로 여러 개의 검색 결과를 하나로 결합

    [Advanced RAG 기법 - 1]

    • Parent Document: 관련된 상위 문서까지 활용해서 검색된 결과에 더 나은 문맥을 제공하고 정보 구조화를 할 수 있는 기법
      • 더 나은 문맥을 고려한 답변을 얻을 수 있지만, 문서의 관계를 유지하고 관리하는 것이 복잡할 수 있고 상위 문서에 대한 추가 처리로 시간이 길어질 수 있음
    • Re-ranker: 초기 검색 결과를 다시 평가하고 재정렬해서 더 나은 검색을 하는 방법
      • 정확성과 관련성을 높일 수 있으나, 추가되는 평가 과정에서 많은 컴퓨팅 자원과 시간이 필요

    [Advanced RAG 기법 - 2]

    • Advanced RAG 기법은 2개 이상 조합하는 방식으로 활용이 가능
    • 예시는 Hybrid-Fusion, Re-ranker, Parent doc 기법을 조합한 Advanced Context Retrieval 기법으로 잘못된 응답을 제거
      • Hybrid-Fusion을 통해 다양한 검색 방법을 결합하고, Re-ranker를 통해 결과를 재정렬하며, Parent Document 기법을 통해 문서의 맥락을 고려하여 정확도를 높임
    • 검색 과정에서 발생할 수 있는 사용자 쿼리와 관련성이 낮은 결과를 효과적으로 제거하고, 사용자에게 더 정확하고 유용한 정보를 제공

    [Advanced RAG를 조합해서 문제를 해결]

    • 문서 하나를 임베딩할 때, 문서 안에 존재하는 이미지, html, 텍스트 등 여러 종류의 데이터를 모두 분리하고 임베딩해서 벡터 스토어에 저장
    • 위 과정을 통해 다양한 형태의 데이터를 통합적으로 검색하고 분석할 수 있게 됨
    • 예시는 차트 이미지를 LLM 모델(Claude3)로 텍스트 변환하고, 변환된 요약 텍스트를 임베딩 모델로 저장하는 과정

    [검색 결과 얻은 관련 컨텐츠]

    • 문서의 의미적 맥락과 키워드 매칭을 모두 고려하여 관련성을 평가한 답변을 얻음

    [Hybrid Search 결과]

    • Parent Document는 상위 문서 컨텍스트를 포함시켜 검색 정확도를 높임
    • HyDE는 주제와 관련된 가상 문서를 생성하고 임베딩하여 의미적 유사성을 극대화

    • Re-ranker는 초기 검색 결과에 대해 다양한 요소를 고려하여 다시 순위를 지정하고, 더 높은 정확도를 얻음
